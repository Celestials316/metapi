# 批量连接测活（账号管理 / API Key 管理）设计

## 背景

当前连接页仅支持单个连接逐个测活。对于账号较多的场景，用户需要频繁点击单条“测活”按钮，效率较低，也难以在移动端快速判断整体可用性。

现有系统已经具备以下基础能力：

- 前端已有单连接测活弹窗：`src/web/pages/accounts/AccountProbeModal.tsx`
- 后端已有真实测活能力：`POST /api/accounts/:id/probe-chat`
- 测活方式已经固定为向上游发送 `hi`，拿到回复文本即视为成功
- 账号页已经分为：
  - `账号管理`
  - `API Key管理`
  - `账号令牌管理`

本设计目标是在不破坏现有单连接测活逻辑的前提下，为连接页增加一个移动端优先、可批量、可实时反馈的“全部测活”能力。

---

## 目标

### 用户目标

- 在 `账号管理` 或 `API Key管理` 页面中，一键发起批量测活
- 支持按“当前分段”或“全部分段”批量测活
- 支持指定一个默认测活模型
- 当某连接没有该模型时，自动回退到该连接的第一个可用且未禁用模型
- 支持选择是否将“未启用”连接纳入结果展示
- 结果页必须实时更新：返回一个显示一个，不等待全部完成
- 移动端必须尽量紧凑，一屏尽量能看到 8~10 条结果

### 产品目标

- 视觉风格必须贴近现有连接页，不引入一套完全不同的设计语言
- 结果页只展示必要信息，不展示用户刚刚选择过的“默认模型”
- 不影响现有单连接测活入口与逻辑

### 非目标

- 不为 `账号令牌管理` 增加批量测活
- 不改动单连接测活的交互流程
- 不引入复杂的后台长任务中心或任务列表页

---

## 用户入口与范围定义

### 入口

在以下页面的“排序与操作”区域新增按钮：

- `账号管理`
- `API Key管理`

按钮文案：

- `全部测活`

### 范围定义

#### 当前分段

表示当前页面当前筛选条件下、当前分段可见的连接集合：

- 在 `账号管理` 分段：仅 Session 连接
- 在 `API Key管理` 分段：仅 API Key 连接

#### 全部分段

表示当前页面当前筛选条件下，以下两个分段的合集：

- `账号管理`
- `API Key管理`

不包含：

- `账号令牌管理`

前端将最终账号 ID 集合直接传给后端，后端不负责解释 UI 分段概念。

---

## 交互设计

### 总体结构

使用 **两步式底部抽屉弹窗**：

1. 配置页
2. 实时结果页

保持与现有 `AccountProbeModal` 一致的交互语义：

- 移动端使用底部抽屉
- 桌面端使用居中弹窗
- 不跳转新页面

---

## Step 1：配置页

### 表单项

#### 1. 测活范围

选项：

- `当前分段`
- `全部分段`

#### 2. 默认测活模型

用户输入或选择一个模型名。

规则：

- 若某连接存在该模型，则优先使用该模型
- 若不存在，则自动回退到该连接第一个可用且未禁用模型

#### 3. 并发数

固定选项：

- `3`
- `4`
- `5`

默认值：

- `4`

#### 4. 未启用连接

开关项：

- 开启：展示并标记“未启用”，但不发起请求
- 关闭：直接跳过，不展示在结果列表中

### 配置页视觉要求

- 延续当前连接页与单连接测活弹窗风格
- 白底 / 浅边框 / 圆角 / 轻渐变提示区
- 控件密度偏紧凑，但不能牺牲点击区域

---

## Step 2：实时结果页

### 展示原则

- **返回一个显示一个**
- 不等待全部完成
- 不预渲染全部连接占位项
- 只展示已经产生结果的条目

### 顶部区域

展示：

- 标题：`测活结果`
- 当前批次进度信息
- 4 个小统计块：
  - 成功
  - 失败
  - 跳过
  - 等待

### 顶部不展示的内容

为保持美观与紧凑，**结果页不展示用户刚刚选过的默认模型**。

原因：

- 用户刚配置过，已知晓默认模型
- 重复展示会占据空间
- 会影响顶部信息密度与美观度

### 结果列表

采用 **紧凑列表（方案 A）**，但进一步吸收“极限紧凑版”的部分密度控制：

- 保留卡片/结果行层次
- 但压缩单条高度
- 目标单条高度约 `50~58px`
- 一屏尽量可见 `8~10` 条

### 单条结果内容

每条只保留最必要的信息：

#### 第一行

- 连接名
- 站点名（如有）
- 状态 / 延迟

#### 第二行

- 实际使用模型
- 若发生回退，显示回退后的模型名

#### 第三行

按状态展示简短文本：

- 成功：显示回复文本第一行
- 失败：显示错误文本第一行
- 未启用：显示“连接已禁用，未发起测活”
- 无模型：显示“该连接没有可用模型”

### 列表排序

最新完成的结果插入列表顶部，方便用户第一时间看到刚返回的条目。

---

## 状态定义

单条结果统一为以下几类状态：

- `success`
- `failed`
- `skipped_disabled`
- `skipped_no_model`

说明：

- `skipped_disabled`：连接被禁用，且用户选择“显示未启用”
- `skipped_no_model`：指定模型不存在，且该连接也没有任何可用且未禁用模型

---

### 文件改动

#### `src/web/pages/Accounts.tsx`

职责：

- 在 `账号管理` / `API Key管理` 的“排序与操作”区增加 `全部测活` 按钮
- 根据当前页筛选状态与分段，生成最终 `accountIds`

#### 新增 `src/web/pages/accounts/BatchAccountProbeModal.tsx`

职责：

- 两步式配置与结果 UI
- 管理测活流的生命周期
- 实时接收结果并渲染
- 支持手动停止

#### `src/web/api.ts`

新增一个支持 **POST + SSE** 的流式请求方法，或者封装专用方法：

- `streamBatchAccountProbe(...)`

需要支持：

- `POST`
- JSON body
- `AbortController`
- 逐条事件回调

#### `src/web/index.css`

新增独立样式命名空间：

- `.batch-account-probe-*`

要求：

- 直接沿用现有变量与弹窗风格
- 不影响已有 `.account-probe-*`

### 前端状态

#### 配置阶段

- `scope: 'segment' | 'all'`
- `preferredModel: string`
- `includeDisabled: boolean`
- `concurrency: 3 | 4 | 5`

#### 结果阶段

- `phase: 'config' | 'running' | 'done'`

  - `failed`
  - `skipped`
  - `pending`
- `running: boolean`

## 后端实现设计

### 路由

在 `src/server/routes/api/accounts.ts` 中新增接口：

- `POST /api/accounts/probe-chat/batch`

返回类型：

- `text/event-stream`

### 服务层

新增服务文件：

- `src/server/services/accountBatchProbeService.ts`

职责：

- 批量调度
- 并发控制
- 单账号 probe 前的模型解析与跳过决策
- 标准化结果输出

### 为什么采用 SSE

本功能属于短时、强交互、用户盯着看的实时操作。

使用 SSE 的优势：

- 可做到“返回一个显示一个”
- 不需要前端轮询
- 比“先建后台任务再查状态”更轻
- 与现有 update-center 的流式日志模式一致，代码风格统一

---

## 批量执行逻辑

### 请求体

- `accountIds: number[]`
- `preferredModel: string`
- `includeDisabled: boolean`
- `concurrency: number`

### 单账号处理顺序

#### 1. 判断连接是否禁用

- 若禁用且 `includeDisabled = true`
  - 返回 `skipped_disabled`
- 若禁用且 `includeDisabled = false`
  - 直接跳过，不出现在结果列表

#### 2. 决定实际测活模型

顺序：

1. 优先使用 `preferredModel`
2. 若不存在，则回退到该连接第一个可用且未禁用模型
3. 若仍不存在，则返回 `skipped_no_model`

#### 3. 执行真实测活

直接复用现有：

- `probeAccountChat({ accountId, modelName })`

测活方式不变：

- 发送固定消息 `hi`

---

## SSE 事件设计

### `event: start`

开始时发送：

- 总账号数
- 实际纳入执行数
- 并发数

### `event: result`

每完成一个账号发送：

- `accountId`
- `accountName`
- `siteName`
- `status`
- `latencyMs`
- `model`
- `usedFallbackModel`
- `message`

说明：

- 成功时 `message` 为回复文本
- 失败时 `message` 为错误文本
- 跳过时 `message` 为跳过原因

### `event: done`

完成时发送：

- 成功数
- 失败数
- 跳过数
- 总耗时

---

## 错误处理

### 前端级

- SSE 连接失败：
  - 弹 toast 提示 `批量测活连接中断`
  - 已收到的结果保留

- 用户手动停止：
  - 中断请求
  - 保留已有结果
  - 顶部状态改为已停止

### 后端级

- 单账号异常：
  - 不影响整个批次
  - 返回一条 `failed`

- 无模型：
  - 返回 `skipped_no_model`

- 禁用账号：
  - 按 `includeDisabled` 决定跳过或展示

---

## 对现有逻辑的影响控制

### 保持不变

- 现有单连接测活入口
- 现有 `probeAccountChat` 核心逻辑
- 现有账号页分段结构

### 新能力只做扩展

- 批量测活仅为单连接测活的调度封装
- 不改代理主链路
- 不改账户调度策略
- 不改模型发现逻辑

## 测试设计

### 前端测试

#### `Accounts.tsx`

- `session` 分段显示 `全部测活`
- `apikey` 分段显示 `全部测活`
- `tokens` 分段不显示
- `全部分段` 只收集 Session + API Key，不包含账号令牌

#### `BatchAccountProbeModal.tsx`

- 配置页可切换范围 / 并发 / 未启用开关
- 开始后进入结果页
- 收到一条 result 即立刻渲染
- `skipped_disabled` 正确显示
- `skipped_no_model` 正确显示
- 停止按钮可中断流
- 结果页不显示默认模型标签

### 后端测试

#### 路由测试

- 非法请求体返回 400
- 返回 `content-type: text/event-stream`
- 能逐条输出 result 事件与最终 done 事件

#### 服务测试

- 优先使用指定模型
- 找不到指定模型时正确回退
- 无可用模型时返回 `skipped_no_model`
- 禁用连接在 `includeDisabled=true` 时返回 `skipped_disabled`
- 单账号失败不阻断其他账号
- 结果按完成顺序流出，而非全部结束后一次返回

## 推荐实现顺序

1. 先补 API 层 POST SSE 能力
2. 实现后端批量测活 SSE 路由与服务
3. 实现批量测活弹窗
4. 在 `Accounts.tsx` 接入入口与账号集合选择
5. 补充前后端测试
6. 再做样式压缩与移动端细调

---

## 最终方案结论

在 `账号管理` 与 `API Key管理` 的“排序与操作”区域新增 `全部测活` 按钮，使用两步式弹窗完成配置与结果展示。后端通过 `POST /api/accounts/probe-chat/batch` 提供 3~5 并发的 SSE 批量测活流，逐条返回实时结果。结果页采用贴近当前连接页风格的紧凑列表，不展示默认模型，只展示必要状态信息与实际结果文本，从而兼顾移动端美观与高信息密度。
